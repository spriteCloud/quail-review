# Examples — what `reviewqa probe` produces

Each subdirectory is the complete output of running

```bash
reviewqa probe --url <site> --dry-run
```

against one of the four reference sites in [`docs/reference-sites.md`](../docs/reference-sites.md).
Generated by the v0.22 binary; no hand-editing.

## Sites

| Directory | URL | Shape |
|---|---|---|
| [`playwright-dev/`](./playwright-dev/) | <https://playwright.dev> | SaaS-shaped docs, JS-heavy |
| [`gohugo-io/`](./gohugo-io/) | <https://gohugo.io> | OSS marketing + content |
| [`books-toscrape-com/`](./books-toscrape-com/) | <https://books.toscrape.com> | E-commerce list-and-detail |
| [`es-wikipedia-org-madrid/`](./es-wikipedia-org-madrid/) | <https://es.wikipedia.org/wiki/Madrid> | Non-English content page |

## What's in each subdirectory

The same shape every probe run produces — a runnable Playwright + Gherkin
project with stakeholder docs alongside:

```
<site>/
├── playwright.config.ts             # defineBddConfig() + multi-project + cross-browser
├── package.json                     # depends on playwright-bdd + @axe-core/playwright
├── tsconfig.json
├── .github/workflows/e2e.yml
└── tests/e2e/
    ├── features/                    # Gherkin contracts (executable via playwright-bdd)
    │   └── <site>-<journey>.feature
    ├── steps/
    │   └── reviewqa.steps.ts        # Given/When/Then → lib/steps.ts
    ├── lib/
    │   └── steps.ts                 # Steps API helper verbs
    ├── api/                         # API contract specs (when site has <form action>)
    │   └── *.api.spec.ts
    ├── a11y/                        # @axe-core/playwright — no serious/critical violations
    │   └── *.a11y.spec.ts
    ├── responsive/                  # mobile / tablet / desktop viewports
    │   └── *.responsive.spec.ts
    ├── perf/                        # page load < SLO (PERF_SLO_MS env)
    │   └── *.perf.spec.ts
    ├── security/                    # baseline security headers (HSTS, CSP, …)
    │   └── *.security.spec.ts
    ├── health/                      # /health, /healthz, /ready, /status probes
    │   └── *.health.spec.ts
    ├── observability/               # x-request-id, server-timing, traceparent
    │   └── *.observability.spec.ts
    ├── contract/                    # OpenAPI-derived contract tests (when /openapi.json exists)
    │   └── *.contract.spec.ts
    ├── i18n/                        # locale switching (when hreflang siblings present)
    │   └── *.i18n.spec.ts
    ├── *-fuzz.spec.ts               # plain Playwright TS fuzz/negative specs
    ├── _fixtures.ts                 # shared test/expect with page-error tracking
    └── docs/
        ├── test-catalogue.md        # stakeholder doc — every journey + priority
        ├── summary.html             # branded HTML deck with priority-mix bar
        └── findings.md              # bug ledger (seeded empty; persists across runs)
```

## Per-site counts (v0.22)

| Site | Files | .feature | a11y | resp | perf | sec | health | obs | api | fuzz |
|---|---|---|---|---|---|---|---|---|---|---|
| `playwright-dev` | 34 | 3 | 5 | 5 | 5 | 1 | 1 | 1 | 1 | 1 |
| `gohugo-io` | 37 | 3 | 5 | 5 | 5 | 1 | 1 | 1 | 0 | 5 |
| `books-toscrape-com` | 32 | 3 | 5 | 5 | 5 | 1 | 1 | 1 | 0 | 0 |
| `es-wikipedia-org-madrid` | 46 | 4 | 5 | 5 | 5 | 1 | 1 | 1 | 8 | 5 |

Total across the matrix: **149 files**, **13 Gherkin journeys**, **20 a11y / 20 responsive / 20 perf** specs, **4 each** of security / health / observability per origin, **9 API contract specs**, **11 fuzz specs**.

OpenAPI (`contract/`) and i18n specs only fire when the site advertises a schema or `hreflang` siblings — none of the four reference sites do, so those directories are absent here. You'll see them light up the moment you point reviewqa at e.g. `https://petstore.swagger.io`.

## Re-generating

These outputs are NOT regenerated by CI — they're snapshots committed for
reference. Refresh after a release with:

```bash
go build -o /tmp/reviewqa ./cmd/reviewqa
rm -rf examples/playwright-dev examples/gohugo-io \
       examples/books-toscrape-com examples/es-wikipedia-org-madrid

for spec in \
  "playwright-dev https://playwright.dev" \
  "gohugo-io https://gohugo.io" \
  "books-toscrape-com https://books.toscrape.com" \
  "es-wikipedia-org-madrid https://es.wikipedia.org/wiki/Madrid"; do
    name=$(echo "$spec" | awk '{print $1}')
    url=$(echo "$spec" | awk '{print $2}')
    mkdir -p "examples/$name"
    /tmp/reviewqa probe --url "$url" --dry-run > "/tmp/$name-probe.log" 2>/dev/null
    # split the `--- path ---` blocks into actual files (see explode helper)
done
```

## Why these four sites?

See [`docs/reference-sites.md`](../docs/reference-sites.md) for the rationale +
legal basis per site. They cover a deliberate spread of shapes — JS-heavy
docs, OSS marketing, e-commerce list/detail, non-English content — so the
generator is exercised against more than just one site's vocabulary.
